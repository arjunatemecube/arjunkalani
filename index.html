Mainframe to Cloud-Native Migration Project Plan (z/VSE to Java/Postgres)

The migration is led by a Principal Solution Architect and aims to replace all 3270 CICS screens with modern web interfaces and move all artifacts (Assembler, COBOL, Copybooks, BMS-maps, JCL with DMS-panels, REXX, EXEC/EXEC2 scripts, XEDIT, TUBES security, VSAM) to their cloud-native equivalents (Java, Bash, PostgreSQL database on AWS RDS).

Data Migration Strategy

Data migration from VSAM files to the PostgreSQL database on Azure is a phased approach to ensure zero data loss and minimal downtime.

Schema Conversion and Validation

Schema Design: Convert the file-based VSAM data structures, including complex record layouts defined in Copybooks, into a relational database schema in PostgreSQL.

Schema Versioning: Use Flyway or Liquibase to manage database schema migrations and ensure the structure remains consistent across all environments (Dev, SIT, UAT, Prod).

Initial Load

  • Extract the complete VSAM data to an intermediate format (e.g., flat files or object storage), perform necessary data type conversions, and load it into PostgreSQL.

Continuous Synchronization (The Cutover)

  • Implement Change Data Capture (CDC) to stream real-time changes from the mainframe VSAM files to the new PostgreSQL database. This allows the legacy system to remain operational while the new environment is being prepared and validated.

Data Quality

  • Implement a Data Watchdog process — a custom Java or script-based utility—to monitor the data streams.
  • Flag Anomalies.
  • Ensure data integrity and consistency between the source and target systems throughout the CDC phase.

Data Masking

  • For non-production environments (SIT, UAT, Dev).
  • Implement Data Masking techniques on sensitive data (e.g., PII, financial details) during the data transfer process to ensure compliance with privacy regulations (a key concern for the security).

Security and Compliance

The Principal Solution Architect must ensure the new platform meets or exceeds the mainframe's legendary reliability and security.

Security

Authentication & Authorization: The new login page must use modern standards (OAuth 2.0/OIDC) for user authentication. Authorization (e.g., checking user privileges for a specific 'location' or Microservice) will be integrated with the Microservices security context. The CISO must approve all security architecture.

Data at Rest/In Transit: Ensure encryption for data at rest (AWS RDS encryption) and in transit (HTTPS for all RESTful APIs).

Non-Functional Requirements (NFRs)

Performance: Define clear performance targets (e.g., API response times, batch job execution windows) that match or improve on the mainframe.

Scalability: Leverage the cloud's elasticity (e.g., auto-scaling Java Microservices, AWS RDS read replicas) to handle peak loads far exceeding the mainframe's capacity.

Disaster Recovery (DR): Implement a robust DR strategy leveraging AWS features like multi-AZ deployment for AWS RDS and automated failover for Microservices, far surpassing traditional mainframe DR plans.

  • The "like-for-like" nature makes UAT crucial for ensuring that established, familiar business processes are not inadvertently altered.
